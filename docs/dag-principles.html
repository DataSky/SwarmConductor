Swarm Conductor DAG 原理详解

有向无环图（DAG）任务编排引擎 · 拓扑调度 · 死锁检测 · 动态扩展

1. 概览

Swarm Conductor 的核心是一个基于有向无环图（DAG）的任务编排引擎。 它让最多 20 个 CodeWhale AI Agent 同时处理一个大型项目，通过声明式任务依赖关系自动协调执行顺序， 避免文件写入冲突，并在 Agent 崩溃或任务失败时自动恢复。

1.1 为什么需要 DAG

在大型项目中，任务之间存在天然的顺序约束：必须先 探索 代码结构，再基于探索结果 制定计划， 然后 实施 修改，最后 审核 和 验证。DAG 模型将这些约束建模为图中的有向边：

  • 节点 (Node)：一个可执行的任务（如分析、编码、审查）
  • 有向边 (Edge)：从任务 A 指向任务 B，表示 B 依赖 A 的完成
  • 无环 (Acyclic)：图中不存在循环依赖，保证任务可以按拓扑序逐一调度

引擎的核心职责是：持续扫描哪些任务的所有依赖已满足，将其标记为就绪，然后分派给空闲的 Agent 执行。

2.3 状态机

每个 TaskNode 的生命周期由 TaskStatus 状态机驱动。状态转换路径如下：

stateDiagram-v2 [*] --> pending: 创建任务 pending --> ready: 无依赖 pending --> blocked: 有未满足的依赖 blocked --> ready: 所有依赖进入终态 ready --> running: 分配 Agent running --> done: 任务成功完成 running --> ready: 失败但可重试 running --> failed: 重试耗尽 running --> interrupted: 死锁中断 done --> [*] failed --> [*] interrupted --> [*] 
// src/dag/types.ts

export type TaskStatus =
  | "pending"     // 等待依赖检查
  | "blocked"     // 存在未满足的依赖
  | "ready"       // 所有依赖已满足，等待 Agent
  | "running"     // 已分配给 Agent 正在执行
  | "done"        // 成功完成
  | "failed"      // 最终失败（重试耗尽）
  | "interrupted" // 被强制中断（死锁解决等）
src/dag/types.ts:11-18
关键设计：done、failed、interrupted 三者统称为终态 (terminal states)。下游任务的解锁条件是「所有依赖均已到达终态」，无论上游成功与否。

3.2 添加任务与依赖边关联

addTasks 批量添加任务时，自动建立双向边引用： 

// src/dag/engine.ts

addTasks(nodes: TaskNode[]): void {
  for (const node of nodes) {
    this.graph.tasks.set(node.id, { ...node })
  }
  // Wire reverse edges — 关键步骤
  for (const node of nodes) {
    for (const depId of node.dependsOn) {
      const dep = this.graph.tasks.get(depId)
      if (!dep) throw new Error(`Task ${node.id} depends on unknown task ${depId}`)
      if (!dep.blocks.includes(node.id)) dep.blocks.push(node.id)
    }
  }
  // 添加后重新计算每个任务的状态
  for (const node of nodes) {
    this.recomputeStatus(node.id)
  }
  this.touch()
}
src/dag/engine.ts:92-109

添加流程：

  1. 将每个任务浅拷贝后存入 tasks Map
  2. 遍历每个任务的 dependsOn，在被依赖任务的 blocks 数组中追加当前任务 ID（这样当被依赖任务完成时，可以快速找到并解锁下游）
  3. 对每个任务调用 recomputeStatus 确定其初始状态（无依赖的 → ready，有依赖的 → blocked）
  4. 更新时间戳

4. 拓扑调度算法

4.1 就绪任务筛选

每次调度时钟滴答时，引擎抓取所有 ready 状态的任务，按 priority 降序排列。 优先级数值越大越靠前。默认优先级为 50，动态插入的任务会获得略高于父任务的优先级。

4.2 调度时钟循环（Tick Loop）

调度循环在 Conductor 类中运行，以固定间隔（默认 500ms）触发：

// src/conductor/index.ts

private async tick(): Promise<void> {
  if (this.approvalGate.hasPending()) return   // 等待人工审批
  this.checkDeadlocks()                         // 先检测死锁

  // 全部任务完成？
  if (this.dag.isComplete()) {
    this.stopScheduler()
    const finalStatus = this.dag.hasCriticalFailure() ? "failed" : "completed"
    this.store.updateRunStatus(finalStatus)
    this.emit(finalStatus === "completed" ? "run.completed" : "run.failed", { phase: this.dag.phase })
    return
  }

  const idleAgents = this.agentMgr.idleInstances()
  if (idleAgents.length === 0) return           // 没有空闲 Agent

  for (const task of this.dag.readyTasks()) {
    if (idleAgents.length === 0) break
    if (this.dag.conflictingRunning(task.scope).length > 0) continue  // 文件冲突，跳过

    // Agent 匹配：按角色优先，其次 general，最后任意空闲
    const agent =
      this.agentMgr.idleByRole(task.role)[0] ??
      this.agentMgr.idleByRole("general")[0] ??
      idleAgents[0]
    if (!agent) continue

    // 文件锁
    if (task.scope.length > 0) {
      if (!this.lockRegistry.tryAcquire(task.scope, agent.id, task.id)) continue
      this.emit("lock.acquired", { agentId: agent.id, taskId: task.id, scope: task.scope })
    }

    idleAgents.splice(idleAgents.indexOf(agent), 1)
    this.dispatch(agent.id, task).catch(err =>
      console.error(`[conductor] dispatch error task=${task.id}:`, err))
  }
}

4.3 Agent 分派（Dispatch）

当调度器选定任务和 Agent 后，执行分派：

  1. 构建上下文：从 SQLite 读取该任务 scope 相关的共享记忆和项目地图，拼接 buildAgentPrompt 生成完整 prompt
  2. 标记状态：Agent 标记为 busy，DAG 中任务从 ready → running
  3. 创建线程：通过 CodeWhale API 创建对话线程并提交 turn
  4. 等待完成：轮询等待 Agent 输出完成（可配置超时、支持流式输出）
  5. 解析输出：解析 ## SUMMARY / CHANGES / EVIDENCE / RISKS / BLOCKERS 五段式输出
  6. 写入记忆：将结果写入 SQLite 共享记忆层，供下游 Agent 使用
  7. 动态扩展：检测输出中的 BLOCKERS 和 RISKS，自动插入后续任务
  8. 审批门：若存在高风险项，暂停调度等候人工决策

5. 环检测与死锁

虽然 DAG 构造时保证无环，但运行时可能出现事实上的死锁：两个 running 任务互相等待对方完成。 系统在每次 tick 时检测并解决。 

// src/dag/engine.ts

/** 返回检测到的环中的任务 ID，无环则返回空数组。 */
detectDeadlock(): string[] {
  const running = this.runningTasks()
  const runningIds = new Set(running.map(t => t.id))
  const cycle: string[] = []

  for (const task of running) {
    if (task.dependsOn.some(depId => runningIds.has(depId))) {
      cycle.push(task.id)
    }
  }
  return cycle
}
src/dag/engine.ts:188-200

在 Conductor 层面，死锁的解决策略是选择环中优先级最低的任务中断：

// src/conductor/index.ts

private checkDeadlocks(): void {
  const cycle = this.dag.detectDeadlock()
  if (cycle.length === 0) return
  this.emit("deadlock.detected", { cycle })
  const victim = cycle
    .map(id => this.dag.getTask(id))
    .filter((t): t is TaskNode => t !== undefined)
    .sort((a, b) => a.priority - b.priority)[0]  // 最低优先级
  if (victim) {
    this.dag.interrupt(victim.id)
    if (victim.assignedTo) {
      this.lockRegistry.releaseByAgent(victim.assignedTo)
      this.agentMgr.markIdle(victim.assignedTo)
    }
  }
}
src/conductor/index.ts:404-419
中断 vs 失败：被中断的任务进入 interrupted 终态，会解锁其下游任务（下游无需等待中断任务完成）。这与 failed 的行为一致：两者都是终态，都会触发 unblockDownstream。

6.2 失败与重试

// src/dag/engine.ts

fail(taskId: string, error: string): void {
  const task = this.mustGet(taskId)
  task.error = error
  task.assignedTo = null

  if (task.retryCount < task.maxRetries) {
    task.retryCount++
    task.error = null
    task.startedAt = null
    this.transition(taskId, "ready")    // 回到就绪，等待重新分派
  } else {
    task.completedAt = Date.now()
    this.transition(taskId, "failed")   // 耗尽重试，进入终态
    this.unblockDownstream(taskId)      // 即使是失败，也解锁下游
  }
}
src/dag/engine.ts:127-142
关键设计决策：即使上游任务失败（重试耗尽），也会解锁下游。这是因为下游任务（如 review）可能仍然有意义——它可以汇报「上游失败，代码未变更」。测试覆盖了这种场景：A (maxRetries=0) → fail → B 被 unblock 进入 ready。

6.4 下游解锁（核心递归传播）

// src/dag/engine.ts

private unblockDownstream(completedId: string): void {
  const task = this.mustGet(completedId)
  for (const blockId of task.blocks) {
    this.recomputeStatus(blockId)
  }
}

private recomputeStatus(id: string): void {
  const task = this.mustGet(id)
  if (task.status !== "pending" && task.status !== "blocked") return

  if (task.dependsOn.length === 0) {
    this.transition(id, "ready")
    return
  }

  // 解锁条件：所有依赖均为终态
  const allTerminal = task.dependsOn.every(depId => {
    const dep = this.graph.tasks.get(depId)
    return dep?.status === "done" || dep?.status === "failed" || dep?.status === "interrupted"
  })

  if (allTerminal) {
    this.transition(id, "ready")
  } else {
    this.transition(id, "blocked")
  }
}
src/dag/engine.ts:278-283, 255-276
注意：解锁只传播一步。如果 A → B → C 且 A 完成，则 B 变为 ready，但 C 仍然 blocked（B 未进入终态）。C 要等到 B 完成后才会被 recompute。测试明确验证了这种多跳链式传播：unblocks through multi-hop dependency chain when dependency fails。

8. 冲突检测

当多个任务同时操作相同文件时会产生写冲突。系统通过两层机制防范：

8.1 DAG 层：Scope 冲突

// src/dag/engine.ts

/** 检查两个 scope 是否存在交集 */
scopesConflict(a: string[], b: string[]): boolean {
  const setA = new Set(a)
  return b.some(f => setA.has(f))
}

/** 查找与给定 scope 冲突的正在运行任务 */
conflictingRunning(scope: string[]): TaskNode[] {
  return this.runningTasks().filter(t => this.scopesConflict(t.scope, scope))
}
src/dag/engine.ts:202-211

在调度 tick 中，如果某个 ready 任务的 scope 与任何 running 任务冲突，该任务会被跳过，等待冲突解除后再分派。

8.2 运行时层：文件锁（FileLockRegistry）

// src/workspace/file-lock.ts

tryAcquire(paths: string[], agentId: string, taskId: string): boolean {
  this.evictExpired()                   // 先驱逐过期锁
  const normalized = paths.map(p => normalizePath(p))
  for (const p of normalized) {
    if (this.locks.has(p)) return false // 任何一个文件已被锁定，则整体失败
  }
  // 全或无：所有文件都锁住才算成功
  const now = Date.now()
  for (const p of normalized) {
    this.locks.set(p, { path: p, heldBy: agentId, taskId,
                         acquiredAt: now, expiresAt: now + this.ttlMs })
  }
  return true
}
src/workspace/file-lock.ts:15-38

文件锁采用原子全或无策略：如果任务声明了多个文件的 scope，必须全部未被锁定才能获取。锁有 TTL（默认 300s），Agent 崩溃后会自动过期释放。

11. 总结

Swarm Conductor 的 DAG 引擎将复杂的多 Agent 协作问题简化为一个有向无环图上的拓扑调度问题。 其核心设计原则：

  • 声明式依赖：通过 dependsOn 显式声明任务间顺序，引擎自动推导就绪和阻塞状态
  • 终态驱动：下游解锁条件是「所有依赖到达终态」，而非「所有依赖成功」——失败不会级联阻塞
  • 优先级调度：就绪任务按 priority 降序分派，动态任务自动获得合理优先级
  • 层层防御：DAG scope 冲突检测 + FileLockRegistry 原子锁 + TTL 过期机制，三重保障文件安全
  • 自愈能力：心跳监控 + 自动重启 + 死锁中断 + SQLite 持久化恢复，系统可在中断后继续执行
  • 动态扩展：Agent 输出驱动新任务插入，让编排从静态计划进化为自适应流程
