关键发现（已修复）
🔴 双重调度竞态条件 已修复
markBusy 和 dag.assign 原本在第一个 await 之后执行，同一 agent 可能在同一 tick 内被分配两次，导致两个任务同时写入相同文件。
src/conductor/index.ts — dispatch() 同步段前移
🔴 SSE 断连默认成功 已修复
连接异常断开时 finalStatus 默认为 "completed"，不完整输出被当作成功结果，下游任务基于错误 context 继续执行。
src/runtime/client.ts — 默认值改为 "failed"
🔴 心跳监控无互斥 已修复
setInterval 调用 check() 没有互斥标志，10+ agents 时 HTTP 健康检查累积延迟超过 interval，堆积并发调用，耗尽文件描述符。
src/conductor/crash-recovery.ts — checking 互斥标志
🔴 addTask 反向边缺失 已修复
单任务版 addTask() 没有填充 blocks[] 反向边，下游任务在依赖完成后永远不会从 blocked 变为 ready，静默死锁。
src/dag/engine.ts — addTask 补充 reverse-edge 写入
🔴 未知依赖 ID 静默失败 已修复
dependsOn 中引用不存在的 task ID 不报错，任务直接进入 blocked 状态且没有任何错误信号，生产环境中极难排查。
src/dag/engine.ts — 添加 throw on unknown dep
🟡 SQLite 标签 LIKE 查询 已修复
memory 表标签用 JSON 列 + LIKE 查询，10K+ 条记录时退化严重。新增 memory_tags 联结表，索引查询 O(1)，同时加 busy_timeout=5000。
src/memory/store.ts — memory_tags join table + 索引
待处理发现（未修复）
严重度 问题 位置 触发条件
中 DAG failed 依赖无恢复路径 — 下游永远 blocked dag/engine.ts 依赖任务重试耗尽后失败
中 restart() 无错误处理 — 崩溃恢复可能无限重试 runtime/agent-manager.ts 进程重启持续失败
中 stopAll() 不等待进程退出 — 端口冲突 runtime/agent-manager.ts 快速重启 conductor
中 CrashRecovery 完全未被测试 tests/ 心跳逻辑任何变更
中 parseTaskOutput() 未被测试 tests/ LLM 格式漂移
低 tick() O(T²) 退化 — runningTasks() 重复调用 conductor/index.ts 1000+ 任务 + 动态插入
